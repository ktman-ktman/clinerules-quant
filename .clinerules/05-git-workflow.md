# Git ワークフロー

## コミットメッセージ形式 (Conventional Commits)

```
<type>: <説明>

<任意の本文>
```

type は次のいずれか: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci`

例:

```
feat: モメンタムシグナルに営業日ラグを追加
fix: 月次リバランスで月末休場日が欠落するバグを修正
test: シャープレシオ計算の境界条件テストを追加
```

## コミット前チェックリスト

- [ ] `pytest` が全件パス
- [ ] `black` / `isort` / `ruff` / `mypy` が通る
- [ ] `bandit -r src/` が警告なし
- [ ] ハードコードされたシークレット・ローカルパスがない
- [ ] `print()` やデバッグコードが残っていない
- [ ] 大きなデータファイル・notebook の出力セルをコミットしていない

## ブランチとPR

- main へ直接コミットしない。ブランチを切って PR を作る
- PR 作成時は最新コミットだけでなく全コミット履歴を確認し、`git diff main...HEAD` で全変更を把握して要約を書く
- PR にはテスト計画(何をどう検証したか)を含める

## クオンツ特有の注意

- notebook (`.ipynb`) はコミット前に出力セルをクリアする(データ・機密の混入防止、diff の可読性)
- データファイル(parquet/CSV)は原則リポジトリに含めない。データはデータ基盤側でバージョン管理する(→ 07-reproducibility.md)
- リサーチ結果を左右するパラメータ変更(ユニバース、期間、コスト前提)はコミットメッセージ本文に理由を書く
