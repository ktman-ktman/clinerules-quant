# 全体原則

## コア原則

### KISS (Keep It Simple)
- 実際に動く最もシンプルな解決策を選ぶ
- 早すぎる最適化を避ける
- 巧妙さより明快さを優先する

### DRY (Don't Repeat Yourself)
- 繰り返されるロジックは共有関数・ユーティリティに抽出する
- コピペによる実装のずれ(drift)を避ける
- 抽象化は「実際に繰り返しが発生してから」導入する。推測で作らない

### YAGNI (You Aren't Gonna Need It)
- 必要になる前に機能や抽象化を作らない
- 投機的な汎用化を避ける。シンプルに始めて、必要になったらリファクタリングする

## イミュータビリティ(重要)

常に新しいオブジェクトを作り、既存オブジェクトを変更(mutate)しない。

```python
# 誤: 引数を破壊的に変更する
def add_fee(position: dict) -> None:
    position["fee"] = position["notional"] * 0.001

# 正: 新しいオブジェクトを返す
def with_fee(position: dict) -> dict:
    return {**position, "fee": position["notional"] * 0.001}
```

理由: 隠れた副作用を防ぎ、デバッグを容易にし、並行処理を安全にする。

## ファイル・関数の構成

- 少数の巨大ファイルより、多数の小さなファイル(高凝集・低結合)
- ファイルは 200〜400 行が目安、最大 800 行
- 関数は 50 行未満に収め、責務を1つに絞る
- 機能・ドメイン単位で構成する(ファイル種別単位ではない)

## コードの臭い(避けるべきパターン)

### 深いネスト
条件分岐が重なり始めたら早期 return を使う(ネスト 4 段以上は禁止)。

```python
# 正: 早期return
def validate_order(order: Order) -> None:
    if order.quantity <= 0:
        raise ValueError("数量は正でなければならない")
    if order.price is None:
        raise ValueError("価格が未設定")
    # メイン処理
```

### マジックナンバー
意味のある閾値・遅延・上限は名前付き定数にする。

```python
MAX_LEVERAGE = 3.0
TRADING_DAYS_PER_YEAR = 252
```

## エラーハンドリング

- すべての階層でエラーを明示的に処理する
- エラーを黙って握りつぶさない(`except Exception: pass` 禁止)
- UI・レポートに出す文言はユーザ向けに、ログには詳細なコンテキストを残す

## 入力バリデーション

- システム境界(ファイル読込、API応答、ユーザ入力)で必ず検証する
- 外部データを信用しない。スキーマ検証(pydantic 等)を活用する
- 明確なエラーメッセージで即座に失敗させる(fail fast)

## 命名規約

- 変数・関数: `snake_case`、内容がわかる名前
- 真偽値: `is_` / `has_` / `should_` / `can_` プレフィックス
- クラス・型: `PascalCase`
- 定数: `UPPER_SNAKE_CASE`

## 完了前チェックリスト

- [ ] コードが読みやすく、命名が適切
- [ ] 関数が小さい(<50行)、ファイルが焦点を保っている(<800行)
- [ ] 深いネストがない(4段未満)
- [ ] エラーハンドリングが明示的
- [ ] ハードコードされた値がない(定数・設定に切り出し)
- [ ] 破壊的変更(mutation)を使っていない
